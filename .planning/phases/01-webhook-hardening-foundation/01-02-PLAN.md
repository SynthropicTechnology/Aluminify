    The function `syncStripeSubscriptionToLocal` does ONE thing: fetches current Stripe subscription state and upserts it locally. Every webhook event type calls this same function.

    Implementation details:
    - Import `getStripeClient` from `@/shared/core/services/stripe.service`
    - Import `getDatabaseClient` from `@/shared/core/database/database`
    - Import `logger` from `@/shared/core/services/logger.service`

    Function signature:
    ```typescript
    export async function syncStripeSubscriptionToLocal(
      stripeSubscriptionId: string,
      metadata?: { empresa_id?: string; plan_id?: string }
    ): Promise<void>
    ```

    Function body (follow Pattern 2 from RESEARCH.md exactly):
    1. Call `stripe.subscriptions.retrieve(stripeSubscriptionId)` to get current Stripe state
    2. Extract period from `stripeSub.current_period_start` and `stripeSub.current_period_end` (these are Unix timestamps, convert to ISO strings)
    3. Resolve plan: get `currentPriceId` from `stripeSub.items.data[0]?.price?.id`, then query `subscription_plans` table with `.or(\`stripe_price_id_monthly.eq.${currentPriceId},stripe_price_id_yearly.eq.${currentPriceId}\`)` to find matching plan
    4. Determine billing interval from `stripeSub.items.data[0]?.price?.recurring?.interval` (default "month")
    5. Get stripe_customer_id as string (handle both string and object Customer)
    6. Build `subscriptionData` object with: stripe_subscription_id, stripe_customer_id, status (cast as SubscriptionStatus), billing_interval, current_period_start (ISO), current_period_end (ISO), cancel_at (ISO or null), canceled_at (ISO or null), plan_id (if resolved), empresa_id (if in metadata)
    7. Check if subscription exists: `.from("subscriptions").select("id, empresa_id").eq("stripe_subscription_id", stripeSubscriptionId).maybeSingle()`
    8. If exists: `.update(subscriptionData).eq("id", existing.id)`
    9. If not exists: validate metadata has empresa_id and plan_id, then `.insert({...subscriptionData, empresa_id, plan_id})`
    10. If not exists and missing metadata: throw Error with message `"Cannot create subscription without empresa_id and plan_id: ${stripeSubscriptionId}"`
    11. Sync `empresas.plano` enum: resolve the `planoMapping` (gratuito->"basico", nuvem->"profissional", personalizado->"enterprise") and update `empresas.plano` for the relevant empresa_id
    12. Log the sync result with `logger.info("billing", "Subscription synced", { stripe_subscription_id, status, empresa_id })`

    Also preserve the helper function `getSubscriptionIdFromInvoice` from the current webhook handler -- extract it into billing.service.ts and export it. This helper handles the Stripe v20+ API change where `invoice.subscription` moved to `invoice.parent.subscription_details.subscription`.

    **NEW POST handler structure (follow Pattern 1 from RESEARCH.md):**

    1. **Rate limiting (IP-based):** Extract client IP from `x-forwarded-for` or `x-real-ip` headers. Call `rateLimitService.checkLimit(\`webhook:${clientIp}\`)`. Return 429 with `{ error: "Too many requests" }` if rate limited.

    2. **Stripe signature verification:** Read raw body via `request.text()`. Call `stripe.webhooks.constructEvent(body, sig, webhookSecret)`. On failure: log with `logger.error`, return 400 with generic message `{ error: "Webhook signature verification failed" }` (do NOT leak the actual error message -- anti-pattern from RESEARCH.md).

    3. **Idempotency guard (processWebhookEvent function):**
       - Check `webhook_events` table: `.from("webhook_events").select("id, status").eq("stripe_event_id", event.id).maybeSingle()`
       - If `existing?.status === "processed"`: log "Duplicate event skipped", return (idempotent no-op)
       - Upsert to 'processing': `.from("webhook_events").upsert({ stripe_event_id: event.id, event_type: event.type, status: "processing", payload: event.data.object as unknown as Json }, { onConflict: "stripe_event_id" }).select("id").single()`
       - Record `startTime = Date.now()`

    4. **Event dispatch:** Based on `event.type`, extract the `stripeSubscriptionId`:
       - `checkout.session.completed`: Get subscription ID from `session.subscription` (string). Extract `empresa_id` and `plan_id` from `session.metadata`. Call `syncStripeSubscriptionToLocal(subId, { empresa_id, plan_id })`.
       - `invoice.paid`, `invoice.payment_failed`: Get subscription ID via `getSubscriptionIdFromInvoice(invoice)`. Call `syncStripeSubscriptionToLocal(subId)`.
       - `customer.subscription.updated`, `customer.subscription.deleted`: Get subscription ID from `subscription.id`. Call `syncStripeSubscriptionToLocal(subId)`.
       - Unknown event type: log warning, skip processing (return 200 -- do NOT return 500 for unknown events).

    5. **Success path:** Update webhook_events to `status: "processed"`, set `processed_at` and `processing_time_ms`.

    6. **Error handling (critical -- follow Pitfall 1 from RESEARCH.md):**
       - Wrap dispatch in try/catch
       - On error: Update webhook_events to `status: "failed"`, set `processing_error: error.message`, `processing_time_ms`
       - Call `Sentry.captureException(error)` for error tracking
       - Distinguish transient vs permanent errors:
         - Transient (DB connection failure, Stripe API timeout): return 500 (Stripe will retry)
         - Permanent (missing metadata, unknown plan): return 200 with logged error (prevent Stripe retry flood)
       - Use heuristic: if error message contains "ECONNREFUSED", "ETIMEDOUT", "503", "network" -> transient (500). Otherwise -> permanent (200).

    7. **Final response:** Return `NextResponse.json({ received: true }, { status: 200 })`

    **Handled event types** (same 5 as before):
    - `checkout.session.completed`
    - `invoice.paid`
    - `invoice.payment_failed`
    - `customer.subscription.updated`
    - `customer.subscription.deleted`
